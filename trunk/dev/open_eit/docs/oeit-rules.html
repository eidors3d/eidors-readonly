Validation Rules

Terminology follows RFC 2119 (http://www.ietf.org/rfc/rfc2119.txt):

MUST This word, or the terms "REQUIRED" or "SHALL", mean that the definition is an absolute requirement of the specification.

 

MUST NOT This phrase, or the phrase "SHALL NOT", mean that the definition is an absolute prohibition of the specification.

SHOULD This word, or the adjective "RECOMMENDED", mean that there may exist valid reasons in particular circumstances to ignore a particular item, but the full implications must be understood and carefully weighed before choosing a different course.

SHOULD NOT This phrase, or the phrase "NOT RECOMMENDED" mean that there may exist valid reasons in particular circumstances when the particular behavior is acceptable or even useful, but the full implications should be understood and the case carefully weighed before implementing any behavior described with this label.

MAY This word, or the adjective "OPTIONAL", mean that an item is truly optional. One vendor may choose to include the item because a particular marketplace requires it or because the vendor feels that it enhances the product while another vendor may omit the same item. An implementation which does not include a particular option MUST be prepared to interoperate with another implementation which does include the option, though perhaps with reduced functionality. In the same vein an implementation which does include a particular option MUST be prepared to interoperate with another implementation which does not include the option (except, of course, for the feature the option provides.)

 

A note on the rule Must not contain text: the rule validates only non-whitespace text since whitespace may occur to make the XML more human readable. Leading and trailing whitespace shall be taken as non-significant within text elements.

xsd namespace is imported from http://www.w3.org/2001/XMLSchema

 

1.                       OEIT File

1.1.                          Must be a valid zip file.

1.3.                          Must contain one oeit.xml file in the root directory.

1.5.                          Must contain a non-empty eit/0000 directory.

62.                    An oeit:conductance must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?(S|Mho(s)?).

63.                    An oeit:conductivity must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?S/([YZEPTGMkdcmunpfazy]|da)?m.

64.                    An oeit:coordinate_system must be a value from the enumeration of {Cartesian, Cylindrical, GPS, Spherical}.

65.                    An oeit:current must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?A.

66.                    An oeit:field_type must be a value from the enumeration of {uint8, uint16, uint32, uint64, int8, int16, int32, int64, float, double}.

67.                    An oeit:force must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?N.

68.                    An oeit:frequency must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?Hz.

69.                    An oeit:impedance must conform to the regular expression pattern: \d+(\.\d+)?\w*([YZEPTGMkdcmunpfazy]|da)?(Ω|Ohm(s)?).
